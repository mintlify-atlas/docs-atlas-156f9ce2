## Overview

The MultiEdit tool types text into multiple input fields in a single operation. It's ideal for filling out forms, entering data into multiple cells, or updating several text fields at once.

## Function Signature

```python
def multi_edit_tool(
    locs: list[list] | None = None,
    labels: list[list] | None = None,
    ctx: Context = None
) -> str
```

## Parameters

<ParamField path="locs" type="list[list]" default="None">
  List of coordinate-and-text triplets. Each element is `[x, y, text]`.
  
  **Example:** `[[100, 200, "John"], [300, 200, "Doe"], [100, 250, "john@example.com"]]`
  
  - `x`, `y`: Coordinates of the input field
  - `text`: Text to type into that field
  
  Either `locs` or `labels` must be provided (not both).
</ParamField>

<ParamField path="labels" type="list[list]" default="None">
  List of label-and-text pairs. Each element is `[label, text]`.
  
  **Example:** `[[15, "John"], [18, "Doe"], [22, "john@example.com"]]`
  
  - `label`: UI element label/ID from a previous Snapshot call
  - `text`: Text to type into that field
  
  **Requirements:** Desktop state must be available (call Snapshot first).
  
  Either `locs` or `labels` must be provided (not both).
</ParamField>

## Return Value

Returns a string confirmation listing all edited elements:

```
Multi-edited elements at: (100,200) with text 'John', (300,200) with text 'Doe', (100,250) with text 'john@example.com'
```

## locs vs labels: When to Use Which

<AccordionGroup>

<Accordion title="Use locs when...">
  **Advantages:**
  - You know the exact field coordinates
  - Working with fixed-position forms
  - No Snapshot call required (faster)
  - Simple, static layouts
  
  **Format:** `[[x1, y1, text1], [x2, y2, text2], ...]`
  
  **Example:** Filling a form that's always in the same position
</Accordion>

<Accordion title="Use labels when...">
  **Advantages:**
  - Field positions may vary
  - Working with dynamic forms
  - Want semantic targeting ("fill the email field")
  - More resilient to UI changes
  
  **Format:** `[[label1, text1], [label2, text2], ...]`
  
  **Requirements:** Must call Snapshot first
  
  **Example:** Filling a web form where field positions change
</Accordion>

</AccordionGroup>

## Parameter Format Differences

| Parameter | Element Format | Example | Notes |
|-----------|---------------|---------|-------|
| `locs` | `[x, y, text]` | `[100, 200, "John"]` | 3 elements: x coordinate, y coordinate, text |
| `labels` | `[label, text]` | `[15, "John"]` | 2 elements: label ID, text |

<Warning>
**Format Validation**

When using `labels`, each item must have exactly 2 elements `[label, text]`. The tool will raise an error if the format is incorrect:

```python
# ❌ Wrong: 3 elements like locs
labels=[[15, 100, "John"]]  # Error!

# ✅ Correct: 2 elements
labels=[[15, "John"]]  # Works!
```
</Warning>

## Best Practices

1. **Call Snapshot first when using labels**
   ```python
   snapshot_tool()  # Populate desktop state
   multi_edit_tool(labels=[[15, "text"], [18, "more text"]])
   ```

2. **Organize data logically**
   ```python
   # Good: Organized by form section
   multi_edit_tool(locs=[
       # Personal info
       [100, 150, "John"],
       [300, 150, "Doe"],
       # Contact info
       [100, 250, "john@example.com"],
       [100, 300, "555-0123"]
   ])
   ```

3. **Handle special characters properly**
   ```python
   # Escape quotes in text
   multi_edit_tool(locs=[
       [100, 200, "She said \"Hello\""],
       [100, 250, "O'Brien"]  # Apostrophes are fine
   ])
   ```

4. **Use consistent tab order**
   - List fields in the order they appear (top to bottom, left to right)
   - This matches natural form flow

## Error Handling

Common errors and solutions:

- **"Either locs or labels must be provided"**: Supply one of the two parameters
- **"Desktop state is empty"**: Call Snapshot before using `labels`
- **"Each label item must be [label, text]"**: Check that each labels element has exactly 2 items
- **"Failed to process label item"**: Label doesn't exist; take a new snapshot
